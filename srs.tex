\section{Introduction}

\subsection{Document Purpose}

This Software Requirements Specification (SRS) document describes the functional and non-functional requirements for \textbf{OpenResearch}, a collaboration-first research platform designed to enable research teams to collaborate in real-time with AI-powered features. This document is intended for developers, stakeholders, QA teams, and project managers involved in the development and maintenance of the system.

The purpose of this document is to:
\begin{itemize}
    \item Define all functional capabilities of the OpenResearch platform
    \item Specify non-functional requirements including performance, security, and quality attributes
    \item Provide a complete reference for system design and implementation
    \item Establish acceptance criteria for testing and validation
    \item Serve as a contractual agreement between stakeholders and development team
\end{itemize}

\subsection{Product Scope}

OpenResearch is a web-based platform that enables:

\begin{itemize}
    \item \textbf{Real-time Collaboration}: Research groups can communicate through live chat with typing indicators
    \item \textbf{AI-Powered Research Assistance}: Integration with LLM models for paper Q\&A, summarization, and discovery
    \item \textbf{Academic Paper Management}: Search, save, organize, and annotate research papers from arXiv
    \item \textbf{Vector-Based Semantic Search}: Find related content across papers using embeddings (pgvector)
    \item \textbf{Report Generation}: Create PDF reports summarizing group research activity
\end{itemize}

The platform consists of three main services:
\begin{enumerate}
    \item \textbf{Frontend Client}: Next.js 16 with React 19
    \item \textbf{Backend Server}: Node.js 20 with Express and Socket.IO
    \item \textbf{AI Service}: Python 3.12 FastAPI with Groq LLM integration
\end{enumerate}

The system benefits research teams by providing a unified platform for collaborative literature review, AI-assisted analysis, and knowledge management, thereby accelerating research workflows and improving team productivity.

\section{Overall Description}

\subsection{Product Overview}

OpenResearch operates as a standalone web application with integration points to external services for AI capabilities and academic paper retrieval. The system architecture consists of three primary tiers:

\begin{enumerate}
    \item \textbf{Presentation Layer}: Next.js/React frontend with real-time WebSocket connections
    \item \textbf{Application Layer}: Node.js backend with Express REST API and Socket.IO server
    \item \textbf{AI Processing Layer}: Python FastAPI service for LLM operations and vector search
    \item \textbf{Data Layer}: PostgreSQL database with pgvector extension
\end{enumerate}

\textbf{External System Dependencies:}
\begin{itemize}
    \item \textit{arXiv API}: Paper search and metadata retrieval
    \item \textit{Groq API}: LLM inference using Llama 3.3 70B model
    \item \textit{SPECTER2 Model}: Local text embedding generation (768 dimensions) --- runs locally without API key
\end{itemize}

The system supports collaborative research workflows where multiple users can simultaneously interact with shared paper collections, conduct AI-assisted analysis, and maintain persistent conversation history. All AI features are explicitly triggered using the \texttt{@ai} prefix to ensure intentional usage and cost control.

\subsection{Product Functionality}

Major system functions include:

\textbf{User Authentication \& Profile Management}
\begin{itemize}
    \item User registration and login (email/password)
    \item JWT-based authentication with refresh tokens
    \item User profile management with research interests
\end{itemize}

\textbf{Research Group Collaboration}
\begin{itemize}
    \item Create and manage research groups
    \item Invite members via email
    \item Role-based access (owner, member)
\end{itemize}

\textbf{Real-time Communication}
\begin{itemize}
    \item Live chat within group sessions
    \item Typing indicators
    \item Message history persistence
\end{itemize}

\textbf{AI-Powered Research Tools}
\begin{itemize}
    \item Paper Q\&A with \texttt{@ai} trigger
    \item Automatic paper summarization with key points extraction
    \item AI-powered paper recommendations
    \item Group AI chat with RAG context
\end{itemize}

\textbf{Paper Management}
\begin{itemize}
    \item Search arXiv for academic papers
    \item Import and save papers to personal/group libraries
    \item Add notes and annotations
\end{itemize}

\textbf{Semantic Search}
\begin{itemize}
    \item Vector-based similarity search across group papers
    \item Find related content using embeddings
\end{itemize}

\textbf{Report Generation}
\begin{itemize}
    \item Generate PDF reports summarizing group activity
    \item Customizable report types (weekly, monthly, custom)
\end{itemize}

    \item \textbf{Database}: PostgreSQL 16+ with pgvector extension
\end{itemize}

\subsection{Design and Implementation Constraints}

\begin{enumerate}
    \item \textbf{AI Trigger Requirement}: All AI features require the \texttt{@ai} trigger to activate, preventing accidental AI calls and controlling costs

    \item \textbf{Group Isolation}: All vector searches and AI context are strictly isolated by group ID to prevent cross-group data leakage

    \item \textbf{Rate Limiting}: API endpoints are rate-limited to prevent abuse:
    \begin{itemize}
        \item Group Chat: 60 requests/minute
        \item Paper Q\&A: 30 requests/minute
        \item Summarization: 10 requests/minute
        \item Report Generation: 5 requests/hour
    \end{itemize}

    \item \textbf{External API Dependencies}:
    \begin{itemize}
        \item Groq API for LLM (Llama 3.3 70B)
        \item arXiv API for paper search
        \item SPECTER2 local model for embeddings (no external dependency)
    \end{itemize}

    \item \textbf{Real-time Communication}: Socket.IO requires persistent WebSocket connections

    \item \textbf{Vector Storage}: Embeddings require approximately 10KB per vector chunk, scaling with paper count
\end{enumerate}

\subsection{Assumptions and Dependencies}

\textbf{Assumptions:}
\begin{itemize}
    \item Users have stable internet connectivity for real-time features
    \item External APIs (Groq, arXiv) maintain availability and API compatibility
    \item SPECTER2 embedding model runs successfully on AI service infrastructure
    \item Users have modern browsers with JavaScript enabled
    \item Research papers are publicly accessible via arXiv
\end{itemize}

\textbf{Dependencies:}
\begin{itemize}
    \item PostgreSQL 16 with pgvector extension for vector storage
    \item Groq API for LLM inference
    \item SPECTER2 model (via Sentence-Transformers) for local embedding generation
    \item arXiv API for paper metadata and search
\end{itemize}
